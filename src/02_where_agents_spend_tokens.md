# 2. 코딩 에이전트는 어디에서 토큰을 쓰는가

코딩 에이전트의 토큰 사용량은 사용자의 질문 길이보다 에이전트가 작업 중 읽고 말하는 양에 더 크게 좌우된다. 특히 도구를 쓰는 에이전트는 모델 입력에 도구 결과가 계속 합쳐진다. `rg` 결과, `sed` 출력, 테스트 실패 로그, package script 목록, git diff, 타입 에러가 모두 다음 판단의 입력이 된다.

## 입력 토큰의 주요 원천

입력 토큰은 대략 다음 순서로 커진다.

- 시스템/개발자 지시: 항상 붙는 기본 규칙과 도구 사용 규칙
- 프로젝트 지시: `AGENTS.md`, repo memory, coding convention
- 사용자 요청: 현재 작업의 목표와 제약
- 대화 이력: 이전 설명, 질문, 중간 보고, 실패와 재시도
- 도구 결과: 파일 내용, 검색 결과, 로그, 테스트 출력, diff
- 에이전트 자체 계획: 이전 단계에서 만든 todo, 가정, 요약

현업에서 가장 위험한 것은 도구 결과다. 시스템 지시와 프로젝트 지시는 길어도 보통 안정적이고 캐시하기 쉽다. 반면 도구 결과는 동적이며, 한 번 잘못 읽으면 수천 줄이 그대로 대화에 들어온다. 로그와 generated file은 토큰 예산을 가장 빠르게 태운다.

## 출력 토큰의 주요 원천

출력 토큰은 코드 생성만이 아니다. 실제로는 다음이 섞인다.

- 분석과 계획
- 사용자 업데이트
- patch 생성
- 테스트 실패 설명
- 최종 보고
- PR 본문 또는 커밋 메시지
- 재시도 시 자기 수정 설명

출력 토큰이 높은 것이 항상 나쁘지는 않다. 큰 코드 변경이나 정교한 마이그레이션에서는 충분한 출력이 필요하다. 문제는 출력이 작업 산출물이 아니라 반복 설명에 쓰일 때다. 예를 들어 사용자가 이미 알고 있는 배경을 매번 요약하거나, 실패 원인을 확정하지 못한 채 가능성을 나열하면 complete가 커지지만 작업은 전진하지 않는다.

## 도구 호출은 무료가 아니다

셸 명령 자체가 모델 토큰은 아니더라도, 명령 결과를 모델이 읽는 순간 비용이 된다. 그래서 "명령을 많이 실행했다"보다 "명령 결과를 얼마나 크게 읽었는가"가 중요하다.

예를 들어 다음 두 작업은 request 수가 같아도 비용이 다르다.

- `rg "functionName" src`로 후보 파일 3개를 찾고, 주변 80줄만 읽는다.
- `cat src/big-file.ts`로 4,000줄을 읽고 그 안에서 직접 찾는다.

두 번째 방식은 탐색을 모델에게 떠넘긴다. 사람은 terminal에서 눈으로 훑고 필요한 부분만 가져올 수 있지만, 에이전트는 출력 전체를 다음 입력으로 보낸다. 그러면 모델은 작업과 무관한 utility, import, type, old code까지 모두 들고 판단해야 한다.

## 재시도가 비용을 지배한다

토큰이 크게 늘어나는 작업은 보통 첫 시도부터 거대한 작업이었던 경우보다, 작은 실패가 여러 번 반복된 경우다. 테스트 실패 로그를 크게 읽고, 원인을 잘못 잡고, 다른 파일을 다시 읽고, 같은 테스트를 다시 돌리는 루프가 누적된다.

따라서 토큰 운영에서 중요한 지표는 총 토큰뿐 아니라 `retry count`다. retry가 늘어날수록 다음 비용도 함께 늘어난다.

- 이전 대화 이력이 길어진다.
- 실패 로그가 누적된다.
- 에이전트가 고려해야 할 가정이 늘어난다.
- 사용자가 정정하거나 추가 설명해야 할 가능성이 커진다.

좋은 에이전트 운영은 실패를 없애는 것이 아니라 실패 출력의 크기를 통제하고, 실패 원인을 빠르게 좁히는 것이다.
