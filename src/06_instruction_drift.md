# 6. 흔들리는 지시와 반복 설명

토큰 낭비는 파일이나 로그에서만 생기지 않는다. 개인 개발자가 매번 같은 말을 다시 설명해야 할 때도 생긴다. 코딩 스타일, 테스트 명령, 금지된 패턴, 리뷰 기준, 배포 제약, 브랜치 규칙이 대화 안에서만 전달되면 세션이 바뀔 때마다 사라진다.

## 지시 흔들림의 증상

지시가 흔들리는 개인 작업 환경에서는 다음 일이 반복된다.

- 사용자가 "이 repo에서는 그렇게 안 해"라고 계속 정정한다.
- 에이전트가 매번 package manager를 잘못 고른다.
- 테스트 명령을 찾느라 `package.json`과 CI 파일을 반복해서 읽는다.
- generated file이나 vendored file을 수정한다.
- 최종 답변 형식이 매번 다르다.
- 리뷰 기준이 작업마다 달라진다.

이 문제는 사용자의 prompt skill 부족이 아니라 지식의 위치 문제다. 자주 쓰는 규칙은 대화가 아니라 repo 안의 안정된 파일에 있어야 한다.

## 지시를 세 층으로 나누기

토큰 효율적인 지시는 세 층으로 나눈다.

- 전역 지시: 모든 프로젝트에서 유지되는 에이전트 운용 원칙
- repo 지시: 이 저장소의 구조, 명령, 금지사항, 검증 규칙
- 작업 지시: 이번 작업의 목표, 범위, 완료 조건

전역 지시와 repo 지시는 캐시 가능한 앞쪽 prefix에 두기 좋다. 작업 지시는 매번 바뀌므로 뒤쪽에 둔다. 이 구분이 없으면 매 요청마다 큰 지시를 새로 만들거나, 동적 작업 지시가 앞쪽 캐시를 깨뜨린다.

## 좋은 지시는 짧은 금지보다 판단 기준이다

나쁜 지시는 "하지 마라"만 늘어놓는다. 좋은 지시는 왜 그런 선택을 해야 하는지 판단 기준을 준다.

예를 들어 다음 지시는 약하다.

```text
큰 파일을 읽지 마라.
```

더 나은 지시는 이렇다.

```text
처음 보는 파일은 먼저 rg로 symbol을 찾고, 관련 함수 주변 80~160줄만 읽는다. 전체 파일 읽기는 검색으로 위치를 좁힐 수 없거나 파일이 300줄 이하일 때만 사용한다.
```

이런 지시는 에이전트가 상황에 맞게 행동할 수 있고, 사용자가 매번 정정할 필요를 줄인다.

## 반복 설명을 없애는 문서

반복 설명을 줄이는 문서는 다음 조건을 만족해야 한다.

- repo 루트에서 쉽게 발견된다.
- 1~2분 안에 읽을 수 있다.
- 명령은 복사 실행 가능한 형태다.
- 금지사항은 이유와 대안이 있다.
- 출력 정책과 검증 정책이 있다.
- 오래된 규칙을 제거하는 유지보수 책임이 있다.

문서가 너무 길면 또 다른 토큰 비용이 된다. 프로젝트 메모리는 "모든 것을 설명하는 문서"가 아니라 "첫 request에서 잘못된 길로 가지 않게 하는 안내판"이어야 한다.
